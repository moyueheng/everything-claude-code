<EXTREMELY-IMPORTANT>
如果你认为某个 skill 有哪怕 1% 的可能性适用于当前任务，你**绝对必须**调用它。

**如果某个 skill 适用于你的任务，你没有选择权。你必须使用它。**

这是不可协商的。这不是可选项。你不能用理性分析来逃避这一点。
</EXTREMELY-IMPORTANT>

## 如何访问 Skills

**在 Claude Code 中：** 使用 `Skill` 工具。当你调用 skill 时，其内容会被加载并呈现给你——直接按照它的指示操作。永远不要使用 Read 工具来读取 skill 文件。

**在其他环境中：** 查阅你所在平台的文档了解如何加载 skills。

# 使用 Skills

## 原则

**在任何回复或行动之前，先调用相关或用户请求的 skills。** 即使只有 1% 的可能性某个 skill 适用，你也应该调用它来确认。如果调用的 skill 最终不适合当前情况，你不需要使用它。

```
用户消息接收
    ↓
可能有 skill 适用？ ──是（哪怕只有1%）──→ 调用 Skill 工具
    ↓ 否                                         ↓
回复（包括澄清问题）                    宣布："使用 [skill] 来 [目的]"
                                              ↓
                                    有检查清单？ ──是──→ 创建 TodoWrite 清单
                                              ↓ 否
                                    严格遵循 skill 执行
                                              ↓
                                    回复（包括澄清问题）
```

## 红旗警告

这些想法意味着**停止**——你在合理化自己的逃避：

| 想法 | 现实 |
|------|------|
| "这只是个简单的问题" | 问题也是任务。检查 skills。 |
| "我需要先获取更多上下文" | Skill 检查在澄清问题之前。 |
| "让我先探索代码库" | Skills 告诉你如何探索。先检查。 |
| "我可以快速查看 git/文件" | 文件缺乏对话上下文。检查 skills。 |
| "让我先收集信息" | Skills 告诉你如何收集信息。 |
| "这不需要正式的 skill" | 如果 skill 存在，就用它。 |
| "我记得这个 skill" | Skills 会演进。读取当前版本。 |
| "这不算是任务" | 行动 = 任务。检查 skills。 |
| "用 skill 太过了" | 简单的事情会变得复杂。用它。 |
| "我先做这一件事" | 在做任何事之前先检查。 |
| "这感觉很有生产力" | 无纪律的行动浪费时间。Skills 能防止这一点。 |
| "我知道这是什么意思" | 知道概念 ≠ 使用 skill。调用它。 |

## Skill 优先级

当多个 skills 可能适用时，按此顺序使用：

1. **先使用流程 skills**（头脑风暴、调试）——这些决定如何着手任务
2. **再使用实现 skills**（前端设计、mcp-builder）——这些指导具体执行

"我们来构建 X" → 先头脑风暴，再实现 skills。
"修复这个 bug" → 先调试，再领域特定的 skills。

## Skill 类型

**严格型**（TDD、调试）：严格遵守。不要偏离既定流程。

**灵活型**（模式）：根据上下文调整原则。

Skill 本身会告诉你它是哪种类型。

## 用户指令

指令说**做什么**，不是**怎么做**。"添加 X" 或 "修复 Y" 并不意味着跳过工作流程。
